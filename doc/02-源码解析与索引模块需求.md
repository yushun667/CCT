# 02 — 源码解析与索引模块细化需求

> 对应需求规范：§3.2 源码解析与索引（FR2.1 ~ FR2.9）  
> 优先级范围：P0 ~ P1

---

## 1. 模块概述

源码解析与索引模块是平台的核心引擎，负责对 C/C++ 源码进行静态分析，提取结构化的符号信息和关系数据，并建立高效的持久化索引，为查询和可视化提供数据基础。**对于远程项目，解析工作由部署在远程服务器上的 Agent（`cct-agent`）执行，解析结果通过 SSH 通道回传到本地。**

### 1.1 职责边界

- 解析 C/C++ 源码文件，生成 AST 并提取语义信息
- 建立符号表（函数、变量、类型、宏）
- 建立关系表（调用关系、包含关系、引用关系）
- 管理索引的持久化存储和更新
- 提供增量解析能力
- 支持并行解析以提升性能
- **支持远程解析：通过远程 Agent 在远程服务器上执行解析，索引数据回传本地**

### 1.2 与其他模块的关系

| 上游模块 | 提供内容 |
|----------|----------|
| 项目管理 | 源码根目录、编译数据库、模块定义、SSH 连接/Agent 状态 |

| 下游模块 | 消费内容 |
|----------|----------|
| 查询与检索 | 符号表、关系表 |
| 可视化分析 | 调用图数据、依赖关系数据 |
| 代码编辑器 | 符号位置信息、悬浮提示数据 |
| 专用分析 | AST 节点、宏信息 |
| AI 智能助手 | 索引数据查询接口 |

## 2. 用例详细描述

### UC2.1 全量解析

| 项目 | 内容 |
|------|------|
| **参与者** | 系统（自动）/ 用户（手动触发） |
| **前置条件** | 项目已创建，源码目录已就绪 |
| **触发方式** | 新建项目后自动提示 / 菜单「分析 → 全量解析」 |
| **优先级** | P0 |

**主流程**：

1. 系统读取项目配置，获取源码根目录和编译数据库路径。
2. 系统扫描源码目录，建立待解析文件列表：
   - 若有编译数据库：以编译数据库中的文件列表为准。
   - 若无编译数据库：扫描 `*.c`、`*.cc`、`*.cpp`、`*.cxx`、`*.h`、`*.hh`、`*.hpp`、`*.hxx` 文件。
3. 系统初始化解析调度器，根据 CPU 核心数创建工作线程池。
4. 调度器将文件分配到工作线程，每个线程：
   a. 使用 libclang（或 tree-sitter）解析源文件生成 AST。
   b. 遍历 AST 提取符号（见 §3.1）。
   c. 遍历 AST 提取关系（见 §3.2）。
   d. 将提取结果写入线程本地缓冲区。
5. 调度器收集各线程结果，合并到全局索引。
6. 系统持久化索引到磁盘（见 §4 数据模型）。
7. 系统发送解析完成事件，携带统计信息（文件数、符号数、关系数、耗时）。

**进度反馈**：

```
ParseProgress {
  total_files: u64           // 待解析文件总数
  parsed_files: u64          // 已解析文件数
  failed_files: u64          // 解析失败文件数
  percentage: f32            // 进度百分比
  current_file: String       // 当前正在解析的文件路径
  symbols_found: u64         // 已发现符号数
  relations_found: u64       // 已发现关系数
  elapsed_seconds: f64       // 已耗时（秒）
  estimated_remaining: f64   // 预计剩余时间（秒）
}
```

**异常流程**：

- **E1**：编译数据库中引用的文件不存在 → 跳过并记录警告，继续解析其他文件。
- **E2**：单文件解析失败（语法错误/编码问题） → 记录错误日志，跳过该文件，继续解析。
- **E3**：内存不足 → 暂停新任务分配，等待已运行任务完成后释放内存。如持续不足，降低并行度后继续。
- **E4**：用户取消解析 → 保存已解析部分的索引，标记为"部分解析"状态。

**验收标准**：

- [ ] 能正确解析标准 C11 / C++17 代码
- [ ] 解析进度实时反馈到前端
- [ ] 解析不阻塞 UI 主线程
- [ ] 解析失败的文件有明确的错误记录
- [ ] 索引数据持久化保存，应用重启后可直接使用

### UC2.2 增量解析

| 项目 | 内容 |
|------|------|
| **参与者** | 系统（自动触发） |
| **前置条件** | 项目已完成至少一次全量解析 |
| **触发方式** | 远程同步完成后自动触发 / 文件变更监听 / 手动触发 |
| **优先级** | P0 |

**主流程**：

1. 系统确定变更文件集合：
   - **远程同步场景**：通过 Git diff 获取变更文件列表（新增/修改/删除）。
   - **本地监听场景**：通过文件系统监听（inotify / FSEvents / ReadDirectoryChanges）检测变更。
   - **手动触发场景**：对比文件修改时间与上次解析时间。
2. 对于每个变更文件：
   - **新增/修改**：重新解析该文件，更新其符号和关系。
   - **删除**：从索引中移除该文件的所有符号和关系。
3. 级联更新：
   - 检查被变更文件包含（`#include`）的头文件是否变更。
   - 若头文件变更，需重新解析所有包含该头文件的源文件。
4. 更新索引并持久化。

**增量解析的粒度**：

- 以文件为最小重解析单元。
- 头文件变更时采用"影响分析"，确定需重解析的源文件范围。
- 使用编译数据库中的依赖信息（若可用）优化影响范围计算。

**验收标准**：

- [ ] 增量解析耗时远小于全量解析（仅解析变更文件及其依赖）
- [ ] 删除文件后索引中不残留该文件的符号数据
- [ ] 头文件修改能正确触发依赖源文件的重解析
- [ ] 增量解析后索引数据一致性不低于全量解析

## 3. 提取规则详述

### 3.1 符号提取（FR2.1）

#### 3.1.1 函数符号

| 属性 | 说明 | 示例 |
|------|------|------|
| `name` | 函数名 | `process_request` |
| `qualified_name` | 完全限定名（含命名空间/类名） | `myns::MyClass::process_request` |
| `file_path` | 定义所在文件 | `src/handler.cpp` |
| `line` | 起始行号 | `42` |
| `column` | 起始列号 | `1` |
| `end_line` | 结束行号 | `85` |
| `is_definition` | 是否为定义（vs 声明） | `true` |
| `return_type` | 返回类型 | `int` |
| `parameters` | 参数列表 `[(type, name)]` | `[("const char*", "path"), ("int", "flags")]` |
| `access` | 访问权限（C++） | `public` / `protected` / `private` |
| `is_virtual` | 是否虚函数 | `true` / `false` |
| `is_static` | 是否静态函数 | `true` / `false` |
| `is_inline` | 是否内联 | `true` / `false` |
| `storage_class` | 存储类 | `extern` / `static` / `none` |
| `body_complexity` | 圈复杂度（可选，P2） | `12` |

#### 3.1.2 变量/常量符号

| 属性 | 说明 |
|------|------|
| `name` | 变量名 |
| `qualified_name` | 完全限定名 |
| `type_name` | 变量类型 |
| `file_path` | 所在文件 |
| `line`, `column` | 位置 |
| `is_definition` | 定义 vs 声明 |
| `scope` | 作用域（global / local / member） |
| `storage_class` | `static` / `extern` / `auto` / `register` |
| `is_const` | 是否常量 |

#### 3.1.3 类型符号

| 属性 | 说明 |
|------|------|
| `name` | 类型名 |
| `kind` | `struct` / `class` / `enum` / `union` / `typedef` |
| `qualified_name` | 完全限定名 |
| `file_path`, `line`, `column` | 位置 |
| `base_classes` | 基类列表（C++继承） |
| `members` | 成员列表引用 |
| `template_params` | 模板参数（C++） |
| `size` | 类型大小（可选，需编译信息） |

#### 3.1.4 宏符号

| 属性 | 说明 |
|------|------|
| `name` | 宏名 |
| `file_path`, `line`, `column` | 定义位置 |
| `is_function_like` | 是否函数式宏 |
| `parameters` | 宏参数列表（函数式宏） |
| `body` | 宏体文本 |
| `expansion` | 展开后内容（可选，P2） |

### 3.2 关系提取

#### 3.2.1 调用关系（FR2.2）

```
CallRelation {
  caller_id: SymbolId           // 调用者函数 ID
  callee_id: SymbolId           // 被调用函数 ID
  call_site_file: String        // 调用发生位置 — 文件
  call_site_line: u32           // 调用发生位置 — 行
  call_site_column: u32         // 调用发生位置 — 列
  is_virtual_dispatch: bool     // 是否虚函数调用
  is_indirect: bool             // 是否间接调用（函数指针）
}
```

**特殊处理**：

- **虚函数调用**：记录为虚调用，同时记录可能的实际目标（基于类层次分析）。
- **函数指针调用**：标记为间接调用，`callee_id` 指向函数指针的类型，并尝试分析赋值点以确定可能目标。
- **宏中的调用**：展开宏后在展开位置记录调用关系。
- **回调函数注册**：识别常见的回调注册模式（如 `register_callback(func_ptr)`），建立注册关系。

#### 3.2.2 包含关系（FR2.3）

```
IncludeRelation {
  source_file: String           // 包含文件（写了 #include 的文件）
  target_file: String           // 被包含文件（被 include 的文件）
  include_line: u32             // #include 指令所在行
  is_system_header: bool        // 是否系统头文件（<> vs ""）
  resolved_path: String         // 解析后的实际文件路径
}
```

#### 3.2.3 引用关系

```
ReferenceRelation {
  symbol_id: SymbolId           // 被引用的符号
  reference_file: String        // 引用所在文件
  reference_line: u32           // 引用所在行
  reference_column: u32         // 引用所在列
  reference_kind: RefKind       // Read | Write | Address | Call | Type
}
```

#### 3.2.4 继承关系（C++）

```
InheritanceRelation {
  derived_class_id: SymbolId    // 派生类
  base_class_id: SymbolId       // 基类
  access: Access                // public | protected | private
  is_virtual: bool              // 是否虚继承
}
```

### 3.3 条件编译处理（FR2.5）

**策略**：

1. **优先使用编译数据库**：从 `compile_commands.json` 中提取每个文件的 `-D` 宏定义和 `-U` 取消定义，作为预处理上下文。
2. **无编译数据库时的降级方案**：
   - 解析所有条件编译分支的符号（保守策略）。
   - 通过常见平台宏推断当前平台（`__linux__`、`_WIN32`、`__APPLE__`）。
   - 用户可在项目配置中手动添加预定义宏。
3. **记录条件编译信息**：符号记录中保存其所在的条件编译上下文，供查询时过滤。

### 3.4 宏处理（FR2.4）

**层次**：

| 层次 | 说明 | 优先级 |
|------|------|--------|
| L1 | 记录宏定义位置和名称 | P0 |
| L2 | 记录宏引用位置 | P0 |
| L3 | 展开函数式宏，记录展开后的调用关系 | P1 |
| L4 | 完整宏展开跟踪（展开链） | P2 |

## 4. 数据模型与存储

### 4.1 索引存储方案

使用嵌入式数据库（推荐 SQLite 或 RocksDB）存储索引数据。

**选型对比**：

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| SQLite | 结构化查询强，易于关联查询 | 大量写入时性能较低 | 符号表、关系表 |
| RocksDB | 高性能写入，适合 KV 存储 | 关联查询需应用层实现 | 大规模数据、快速索引 |
| 混合方案 | 两者优势互补 | 复杂度增加 | 推荐方案 |

**推荐：混合方案**

- SQLite：存储符号元数据和关系表（支持复杂查询）
- 内存索引（如 Trie / 倒排索引）：存储符号名搜索索引（支持模糊搜索）
- 文件缓存：存储 AST 序列化数据（支持增量更新）

### 4.2 数据库表结构（SQLite）

```sql
-- 符号表
CREATE TABLE symbols (
  id          INTEGER PRIMARY KEY,
  name        TEXT NOT NULL,
  qualified_name TEXT NOT NULL,
  kind        TEXT NOT NULL,  -- function | variable | type | macro
  sub_kind    TEXT,           -- struct | class | enum | union | typedef (for types)
  file_path   TEXT NOT NULL,
  line        INTEGER NOT NULL,
  column      INTEGER NOT NULL,
  end_line    INTEGER,
  is_definition BOOLEAN NOT NULL DEFAULT 0,
  return_type TEXT,
  parameters  TEXT,           -- JSON array
  access      TEXT,           -- public | protected | private
  attributes  TEXT,           -- JSON object for extra attributes
  project_id  TEXT NOT NULL
);

CREATE INDEX idx_symbols_name ON symbols(name);
CREATE INDEX idx_symbols_qualified ON symbols(qualified_name);
CREATE INDEX idx_symbols_file ON symbols(file_path);
CREATE INDEX idx_symbols_kind ON symbols(kind);

-- 调用关系表
CREATE TABLE call_relations (
  id          INTEGER PRIMARY KEY,
  caller_id   INTEGER NOT NULL REFERENCES symbols(id),
  callee_id   INTEGER NOT NULL REFERENCES symbols(id),
  file_path   TEXT NOT NULL,
  line        INTEGER NOT NULL,
  column      INTEGER NOT NULL,
  is_virtual  BOOLEAN NOT NULL DEFAULT 0,
  is_indirect BOOLEAN NOT NULL DEFAULT 0
);

CREATE INDEX idx_calls_caller ON call_relations(caller_id);
CREATE INDEX idx_calls_callee ON call_relations(callee_id);

-- 包含关系表
CREATE TABLE include_relations (
  id          INTEGER PRIMARY KEY,
  source_file TEXT NOT NULL,
  target_file TEXT NOT NULL,
  line        INTEGER NOT NULL,
  is_system   BOOLEAN NOT NULL DEFAULT 0,
  resolved_path TEXT
);

CREATE INDEX idx_includes_source ON include_relations(source_file);
CREATE INDEX idx_includes_target ON include_relations(target_file);

-- 引用关系表
CREATE TABLE reference_relations (
  id          INTEGER PRIMARY KEY,
  symbol_id   INTEGER NOT NULL REFERENCES symbols(id),
  file_path   TEXT NOT NULL,
  line        INTEGER NOT NULL,
  column      INTEGER NOT NULL,
  ref_kind    TEXT NOT NULL  -- read | write | address | call | type
);

CREATE INDEX idx_refs_symbol ON reference_relations(symbol_id);
CREATE INDEX idx_refs_file ON reference_relations(file_path);

-- 继承关系表
CREATE TABLE inheritance_relations (
  id          INTEGER PRIMARY KEY,
  derived_id  INTEGER NOT NULL REFERENCES symbols(id),
  base_id     INTEGER NOT NULL REFERENCES symbols(id),
  access      TEXT NOT NULL,
  is_virtual  BOOLEAN NOT NULL DEFAULT 0
);

-- 文件信息表（用于增量解析）
CREATE TABLE file_info (
  file_path     TEXT PRIMARY KEY,
  last_modified INTEGER NOT NULL,  -- 文件修改时间戳
  content_hash  TEXT NOT NULL,     -- 文件内容 SHA-256 哈希
  parse_status  TEXT NOT NULL,     -- success | failed | skipped
  error_message TEXT,
  symbol_count  INTEGER NOT NULL DEFAULT 0,
  parse_time_ms INTEGER            -- 解析耗时（毫秒）
);
```

## 5. 并行解析架构（FR2.8）

### 5.1 架构设计

```
┌──────────────────────────────────────────────────────┐
│                    解析调度器                          │
│  ┌──────────┐  ┌──────────────┐  ┌────────────────┐  │
│  │ 文件队列  │  │ 依赖分析器   │  │ 结果收集器     │  │
│  └──────────┘  └──────────────┘  └────────────────┘  │
└──────────────────────────────────────────────────────┘
        │              │                    ▲
        ▼              ▼                    │
┌───────────────────────────────────────────┤
│            工作线程池                      │
│  ┌────────┐ ┌────────┐ ┌────────┐        │
│  │Worker 1│ │Worker 2│ │Worker N│  ──────┘
│  │        │ │        │ │        │
│  │ clang  │ │ clang  │ │ clang  │
│  │ index  │ │ index  │ │ index  │
│  └────────┘ └────────┘ └────────┘
└───────────────────────────────────────────┘
```

### 5.2 并行策略

| 策略 | 说明 |
|------|------|
| **线程池大小** | 默认 = CPU 逻辑核心数 - 1（预留 1 核给 UI 和系统），用户可手动配置 |
| **任务分配** | 按文件粒度分配，每个工作线程独立解析一个源文件 |
| **头文件共享** | 头文件的 AST 可缓存共享（PCH 预编译头），减少重复解析 |
| **结果合并** | 各线程将结果写入线程本地 buffer，完成后批量合并到全局索引 |
| **错误隔离** | 单文件解析失败不影响其他文件，错误记录到日志 |

### 5.3 数据一致性保障

- 符号 ID 使用全局原子计数器分配，避免冲突。
- 关系数据引用符号 ID，在合并阶段统一解析符号引用。
- 索引写入使用事务（SQLite）或写锁（RocksDB），保证原子性。
- 合并完成后执行一致性校验（引用完整性检查）。

### 5.4 内存管理

- 每个工作线程限制最大内存使用量（通过 AST 节点数估算）。
- 解析完成后立即释放 AST 对象，仅保留提取的结构化数据。
- 大文件（> 10 万行）单独分配更多内存配额，必要时降低并行度。
- 全局内存监控，达到阈值时暂停新任务分配。

## 6. 远程解析架构（FR2.9）

### 6.1 架构概览

对于远程项目，解析不在本地执行，而是由部署在远程服务器上的 `cct-agent` 执行：

```
┌────────────────────┐         SSH 隧道           ┌────────────────────────┐
│   桌面端（本地）     │◄═══════════════════════════►│   远程服务器            │
│                    │                             │                        │
│  ┌──────────────┐  │   1. 发送解析请求            │  ┌──────────────────┐  │
│  │ 项目管理     │──┼──────────────────────────►──┤  │   cct-agent      │  │
│  └──────────────┘  │                             │  │                  │  │
│                    │   2. 进度事件流               │  │  ┌────────────┐ │  │
│  ┌──────────────┐  │◄─────────────────────────────┤  │  │ 解析引擎   │ │  │
│  │ 进度显示     │  │                             │  │  │ libclang   │ │  │
│  └──────────────┘  │   3. 回传索引数据库           │  │  │ tree-sitter│ │  │
│                    │◄─────────────────────────────┤  │  └────────────┘ │  │
│  ┌──────────────┐  │      (压缩传输)              │  │                  │  │
│  │ 本地索引存储  │  │                             │  │  ┌────────────┐ │  │
│  │ (SQLite)    │  │                             │  │  │ 远程源码   │ │  │
│  └──────────────┘  │                             │  │  │ /path/to/  │ │  │
│                    │                             │  │  │ source/    │ │  │
│  ┌──────────────┐  │                             │  │  └────────────┘ │  │
│  │ 查询/可视化  │  │                             │  └──────────────────┘  │
│  └──────────────┘  │                             │                        │
└────────────────────┘                             └────────────────────────┘
```

### 6.2 远程解析流程

**全量远程解析**：

1. 桌面端通过 SSH 向远程 Agent 发送解析请求：
   ```
   RemoteParseRequest {
     source_root: String           // 远程源码根目录
     compile_db_path: Option<String> // 远程编译数据库路径
     max_threads: u32              // 最大并行线程数
     max_memory_mb: u64            // 内存限制
   }
   ```
2. 远程 Agent 接收请求后：
   a. 扫描远程源码目录，建立文件列表。
   b. 初始化解析引擎（libclang / tree-sitter）。
   c. 使用多线程并行解析所有源文件（利用远程服务器的 CPU 资源）。
   d. 将结果写入临时 SQLite 数据库。
   e. 持续通过 SSH 通道发送进度事件。
3. 解析完成后，Agent 压缩索引数据库（zstd 压缩）。
4. 通过 SSH/SFTP 通道将压缩后的索引数据传输到桌面端。
5. 桌面端解压并加载索引数据到本地 SQLite。
6. 后续查询和可视化操作基于本地索引执行，无需再次访问远程。

**增量远程解析**：

1. 桌面端请求远程 Agent 检测文件变更（基于文件修改时间/哈希对比）。
2. Agent 仅重解析变更的文件。
3. Agent 生成增量索引补丁（变更的符号/关系）。
4. 增量补丁传输回桌面端并合并到本地索引。

### 6.3 桌面端与 Agent 通信协议

通信基于 SSH 通道上的 JSON-RPC 2.0 协议：

**请求格式**：
```json
{
  "jsonrpc": "2.0",
  "method": "parse/start",
  "params": {
    "source_root": "/home/user/linux-kernel",
    "compile_db_path": "/home/user/linux-kernel/compile_commands.json",
    "max_threads": 8
  },
  "id": 1
}
```

**进度通知**（Agent → 桌面端）：
```json
{
  "jsonrpc": "2.0",
  "method": "parse/progress",
  "params": {
    "total_files": 28000,
    "parsed_files": 5600,
    "failed_files": 3,
    "percentage": 20.0,
    "current_file": "kernel/sched/core.c",
    "symbols_found": 125000,
    "elapsed_seconds": 180.5
  }
}
```

**完成响应**：
```json
{
  "jsonrpc": "2.0",
  "result": {
    "status": "completed",
    "index_file": "/tmp/cct-agent/index-20260301.db.zst",
    "index_size_bytes": 52428800,
    "compressed_size_bytes": 15728640,
    "total_symbols": 850000,
    "total_relations": 2300000,
    "elapsed_seconds": 1200.5
  },
  "id": 1
}
```

**Agent 支持的 RPC 方法**：

| 方法名 | 说明 |
|--------|------|
| `parse/start` | 启动全量解析 |
| `parse/incremental` | 启动增量解析（传入变更文件列表或自动检测） |
| `parse/cancel` | 取消正在进行的解析 |
| `parse/status` | 查询当前解析状态 |
| `index/transfer` | 请求传输索引数据 |
| `index/diff` | 请求增量索引补丁 |
| `file/list` | 列出目录文件 |
| `file/read` | 读取文件内容 |
| `file/stat` | 获取文件信息 |
| `system/info` | 获取服务器系统信息 |
| `agent/version` | 获取 Agent 版本信息 |
| `agent/shutdown` | 关闭 Agent |

### 6.4 索引数据传输优化

| 优化措施 | 说明 |
|----------|------|
| **zstd 压缩** | SQLite 文件使用 zstd 算法压缩，压缩率约 60-70% |
| **增量传输** | 增量解析后仅传输变更部分，而非整个索引 |
| **SSH 复用** | 使用 SSH ControlMaster 复用已有连接 |
| **流式传输** | 大索引文件分块传输，支持断点续传 |
| **并行传输** | 可使用多个 SSH 通道并行传输（P2） |

### 6.5 远程 Agent 内部架构

```
cct-agent (Rust 二进制)
├── main.rs                     // 入口，启动 RPC 服务器
├── rpc/                        // JSON-RPC 处理
│   ├── server.rs               // stdin/stdout RPC 服务（通过 SSH）
│   └── handlers.rs             // 请求处理器
├── parser/                     // 解析引擎（与桌面端共享代码）
│   ├── clang_parser.rs
│   ├── ts_parser.rs
│   └── scheduler.rs
├── indexer/                    // 索引生成
│   ├── database.rs
│   └── compressor.rs
└── utils/
    ├── logging.rs
    └── system_info.rs
```

Agent 与桌面端的解析引擎共享核心代码（通过 Rust workspace 的共享 crate），确保解析结果一致性。

## 7. 接口设计

| 命令名 | 参数 | 返回值 | 说明 |
|--------|------|--------|------|
| `start_full_parse` | `project_id` | — | 启动全量解析（本地或远程，异步） |

| `parse-error` | `ParseError` | 单文件解析错误 |
| `index-transfer-progress` | `TransferProgress` | 索引数据回传进度（仅远程） |

## 8. 解析引擎选型

### 7.1 候选方案对比

| 方案 | 精确度 | 性能 | 易用性 | 维护成本 |
|------|--------|------|--------|----------|
| **libclang (C API)** | 高（完整语义分析） | 中等 | Rust 绑定可用 | 低 |
| **clang AST (C++ API)** | 最高 | 中等 | 需 C++ 集成 | 中 |
| **tree-sitter** | 中（语法级，无语义） | 高 | Rust 原生支持 | 低 |
| **混合方案** | 高 | 高 | 复杂 | 中 |

### 7.2 推荐方案：libclang 为主 + tree-sitter 辅助

- **libclang**：用于主要的符号提取和关系分析（依赖编译数据库获取完整语义）。
- **tree-sitter**：用于无编译数据库时的降级解析（语法级符号提取），以及快速语法高亮。
- 通过 Rust 的 `clang-sys` crate 调用 libclang。

## 9. 错误码定义

| 错误码 | 说明 |
|--------|------|
| `E_PARSE_NO_SOURCE` | 源码目录为空或无 C/C++ 文件 |
| `E_PARSE_CLANG_INIT` | libclang 初始化失败 |
| `E_PARSE_FILE_READ` | 文件读取失败 |
| `E_PARSE_SYNTAX` | 语法解析错误 |
| `E_PARSE_ENCODING` | 文件编码不支持 |
| `E_PARSE_OOM` | 内存不足 |
| `E_PARSE_CANCELLED` | 解析被用户取消 |
| `E_INDEX_WRITE` | 索引写入失败 |
| `E_INDEX_CORRUPT` | 索引数据损坏 |
| `E_REMOTE_AGENT_UNREACHABLE` | 远程 Agent 不可达 |
| `E_REMOTE_AGENT_PARSE_FAILED` | 远程 Agent 解析失败 |
| `E_REMOTE_TRANSFER_FAILED` | 索引数据传输失败 |
| `E_REMOTE_TRANSFER_INTERRUPTED` | 索引数据传输中断 |
| `E_REMOTE_DECOMPRESS_FAILED` | 索引数据解压失败 |
