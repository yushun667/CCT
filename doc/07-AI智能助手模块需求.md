# 07 — AI 智能助手模块细化需求

> 对应需求规范：§3.7 AI 智能助手（FR7.1 ~ FR7.7）  
> 优先级范围：P0 ~ P2

---

## 1. 模块概述

AI 智能助手模块为平台注入智能化能力，通过自然语言交互帮助用户理解代码架构、查询调用关系、分析依赖等。它集成了大语言模型（LLM）、技能调用（Function Calling）、多 Agent 协同和 MCP 协议支持。

### 1.1 职责边界

- 提供自然语言问答界面
- 管理对话上下文（当前文件、符号、项目）
- 调用平台内部功能作为 AI 技能（Tools/Functions）
- 支持多 Agent 协同处理复杂任务
- 实现 MCP 服务端/客户端
- 管理对话历史

### 1.2 与其他模块的关系

| 模块 | 交互方式 |
|------|----------|
| 查询与检索 | AI 通过技能调用查询符号、调用关系等 |
| 可视化分析 | AI 可请求生成/更新图视图 |
| 代码编辑器 | AI 感知当前编辑器上下文 |
| 项目管理 | AI 感知当前项目信息 |
| 专用分析 | AI 调用内核/IPC 分析技能 |

## 2. 用例详细描述

### UC7.1 自然语言问答

| 项目 | 内容 |
|------|------|
| **参与者** | 所有用户角色 |
| **触发方式** | 侧边栏 AI 聊天面板 / 快捷键 `Ctrl+L` |
| **优先级** | P0 |

**主流程**：

1. 用户在聊天输入框中输入自然语言问题。
2. 系统收集上下文信息：
   - 当前项目名称和类型
   - 当前编辑器打开的文件和光标位置（如有）
   - 当前图视图中选中的节点（如有）
   - 用户手动指定的分析范围
3. 系统将用户问题 + 上下文 + 对话历史组装为 LLM 请求。
4. LLM 分析问题，决定是否需要调用技能：
   - **无需技能**：直接生成回答（如解释概念、代码阅读建议）。
   - **需要技能**：生成技能调用请求（如查询函数调用者）。
5. 若需技能调用：
   a. 系统执行技能调用，获取结果。
   b. 将技能结果反馈给 LLM。
   c. LLM 基于技能结果生成最终回答。
   d. 可能需要多轮技能调用（迭代推理）。
6. 系统将回答以流式方式显示在聊天面板中。
7. 回答中包含的代码引用可点击跳转。

**典型问答示例**：

| 用户问题 | AI 行为 | 示例回答 |
|----------|---------|----------|
| "main 函数调用了哪些函数？" | 调用 `query_callees("main")` 技能 | "main 函数调用了以下 5 个函数：init_system()、load_config()..." |
| "从 sys_read 到 ext4_file_read 的调用路径是什么？" | 调用 `query_call_path` 技能 | "从 sys_read 到 ext4_file_read 存在以下路径：sys_read → vfs_read → ..." |
| "网络模块和文件系统模块之间有哪些依赖？" | 调用 `query_module_deps` 技能 | "网络模块依赖文件系统模块的以下函数：..." |
| "这个函数的作用是什么？" | 读取当前文件和函数上下文 | 基于函数代码和调用关系进行解释 |
| "帮我分析这个项目的整体架构" | 调用多个技能（模块列表、依赖图、关键函数） | 综合分析报告 |

**流式响应**：

```
StreamChunk {
  type: "text" | "code" | "skill_call" | "skill_result" | "error"
  content: String
  metadata: Option<ChunkMeta>
}

ChunkMeta {
  skill_name: Option<String>      // 正在调用的技能名
  file_reference: Option<FileRef> // 代码引用（可点击跳转）
  progress: Option<f32>           // 处理进度
}
```

**验收标准**：

- [ ] 用户问题能得到基于项目数据的准确回答
- [ ] AI 回答以流式方式逐字显示
- [ ] 回答中的代码引用可点击跳转到编辑器
- [ ] 技能调用过程对用户透明（显示"正在查询调用关系..."等提示）
- [ ] 单轮回答响应开始时间 < 3 秒

### UC7.2 上下文感知

| 项目 | 内容 |
|------|------|
| **优先级** | P0 |

**自动上下文收集**：

| 上下文来源 | 收集内容 | 收集时机 |
|------------|----------|----------|
| 当前项目 | 项目名、类型、模块列表 | 对话开始时 |
| 编辑器 | 当前文件路径、光标所在函数 | 每次提问时 |
| 编辑器选区 | 用户选中的代码片段 | 提问时有选区 |
| 图视图 | 选中的节点列表 | 提问时有选中 |
| 对话历史 | 最近 N 轮对话 | 每次提问时 |

**手动范围指定**：

用户可通过 UI 控件指定分析范围：

```
AnalysisScope {
  level: ScopeLevel               // Project | Module | Directory | File
  target: String                   // 模块名/目录路径/文件路径
}
```

**上下文窗口管理**：

- 对话历史按 token 数量管理，超出窗口时进行摘要压缩。
- 技能调用结果缓存，避免相同查询重复执行。
- 大段代码上下文使用摘要 + 关键片段的方式传递，避免超出 token 限制。

**验收标准**：

- [ ] AI 能感知当前打开的文件和光标位置
- [ ] AI 能感知编辑器中选中的代码
- [ ] 用户可手动指定分析范围
- [ ] 上下文变化后 AI 回答能反映最新状态

### UC7.3 技能调用（Function Calling）

| 项目 | 内容 |
|------|------|
| **优先级** | P0 |

#### 2.3.1 技能定义格式

遵循 OpenAI Function Calling 格式：

```json
{
  "name": "search_symbols",
  "description": "在当前项目中搜索符号（函数、变量、类型、宏）。使用此技能查找代码中的特定元素。",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜索关键字，支持模糊匹配"
      },
      "kind": {
        "type": "string",
        "enum": ["function", "variable", "type", "macro", "all"],
        "description": "符号类型过滤",
        "default": "all"
      },
      "limit": {
        "type": "integer",
        "description": "最大返回结果数",
        "default": 10
      }
    },
    "required": ["query"]
  }
}
```

#### 2.3.2 内置技能清单

| 技能名 | 说明 | 对应查询接口 | 优先级 |
|--------|------|-------------|--------|
| `search_symbols` | 搜索符号 | `query_symbols` | P0 |
| `get_callers` | 获取函数调用者 | `query_callers` | P0 |
| `get_callees` | 获取函数被调用者 | `query_callees` | P0 |
| `find_call_path` | 查找两函数间调用路径 | `query_call_path` | P0 |
| `get_references` | 获取符号引用 | `query_references` | P0 |
| `get_file_includes` | 获取文件包含关系 | `query_includes` | P1 |
| `get_module_deps` | 获取模块依赖 | `get_module_graph` | P1 |
| `read_source_code` | 读取指定文件的源码 | `read_file` | P0 |
| `get_function_info` | 获取函数详细信息 | `get_symbol_info` | P0 |
| `list_syscalls` | 列出系统调用 | `list_syscalls` | P1 |
| `get_ipc_services` | 获取 IPC 服务列表 | `list_ipc_services` | P1 |
| `show_call_graph` | 在图视图中显示调用图 | 前端事件 | P1 |
| `navigate_to_code` | 在编辑器中跳转到代码 | 前端事件 | P1 |

#### 2.3.3 技能执行流程

```
用户提问 → LLM 分析 → [生成技能调用]
                              ↓
                     执行技能函数 → 获取结果
                              ↓
                     结果反馈 LLM → [可能再次调用技能]
                              ↓
                     生成最终回答 → 展示给用户
```

**多步技能调用**：

- LLM 可在一次推理中请求调用多个技能（并行执行）。
- LLM 可基于前一步技能结果决定后续技能调用（串行推理）。
- 最大技能调用次数限制：单次对话 10 次（可配置）。

**验收标准**：

- [ ] LLM 能正确选择并调用合适的技能
- [ ] 技能调用结果正确反馈给 LLM
- [ ] 支持多步技能调用（迭代推理）
- [ ] 技能调用过程有用户可见的状态提示

### UC7.4 多 Agent 协同

| 项目 | 内容 |
|------|------|
| **优先级** | P2 |

#### 2.4.1 Agent 架构

```
┌──────────────────────────────────────────────┐
│                 协调器（Orchestrator）          │
│  接收用户请求，拆解任务，分配给合适的 Agent     │
└──────────────────┬───────────────────────────┘
                   │
    ┌──────────────┼──────────────┐
    ▼              ▼              ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 规划Agent │  │ 执行Agent │  │ 验证Agent │
│          │  │          │  │          │
│ 分析问题  │  │ 调用技能  │  │ 检查一致  │
│ 制定计划  │  │ 获取数据  │  │ 性和准确  │
│ 拆解子任  │  │ 执行分析  │  │ 性       │
│ 务       │  │          │  │          │
└─────────┘  └─────────┘  └─────────┘
```

#### 2.4.2 Agent 类型

| Agent | 职责 | 使用场景 |
|-------|------|----------|
| **规划 Agent** | 分析复杂问题，拆解为子任务，制定执行计划 | 复杂架构分析请求 |
| **执行 Agent** | 调用平台技能获取数据，执行具体分析 | 调用查询、路径搜索 |
| **验证 Agent** | 检查其他 Agent 的输出，验证一致性和准确性 | 交叉验证分析结果 |
| **总结 Agent** | 将多个 Agent 的输出整合为用户可读的最终报告 | 生成分析报告 |

#### 2.4.3 Agent 通信协议

```
AgentMessage {
  from_agent: AgentId
  to_agent: AgentId
  message_type: MessageType       // TaskAssignment | Result | Query | Validation
  content: String                 // 消息内容
  attachments: Vec<Attachment>    // 附件（数据、图表等）
  timestamp: DateTime
}
```

**执行模式**：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **串行** | Agent 按顺序执行，前一个完成后下一个开始 | 依赖前序结果 |
| **并行** | 多个 Agent 同时执行独立子任务 | 子任务间无依赖 |
| **管道** | 前一个 Agent 的输出作为后一个的输入 | 数据加工链 |

**验收标准**：

- [ ] 复杂问题能被正确拆解为子任务
- [ ] 多个 Agent 能协同完成分析
- [ ] Agent 间通信机制可靠
- [ ] 最终结果整合为连贯的报告

### UC7.5 MCP 协议支持

| 项目 | 内容 |
|------|------|
| **优先级** | P1 |

#### 2.5.1 MCP 服务端

平台作为 MCP 服务端，暴露以下资源和工具：

**资源（Resources）**：

| 资源 URI | 说明 | MIME 类型 |
|----------|------|-----------|
| `cct://project/info` | 当前项目基本信息 | `application/json` |
| `cct://project/modules` | 模块列表和定义 | `application/json` |
| `cct://symbols/{id}` | 符号详细信息 | `application/json` |
| `cct://files/{path}` | 源文件内容 | `text/plain` |
| `cct://graph/call` | 调用图数据 | `application/json` |
| `cct://graph/module-deps` | 模块依赖图数据 | `application/json` |

**工具（Tools）**：

与技能清单（§2.3.2）保持一致，MCP 工具名和参数格式与技能定义相同。

**传输方式**：

- **stdio**：通过标准输入/输出与本地 MCP 客户端通信（默认）。
- **HTTP/SSE**：通过 HTTP Server-Sent Events 与远程 MCP 客户端通信。

#### 2.5.2 MCP 客户端

平台作为 MCP 客户端，可连接外部 MCP 服务获取额外上下文：

| 外部服务示例 | 提供内容 |
|-------------|----------|
| 技术文档服务 | API 文档、标准文档 |
| 代码搜索服务 | 代码片段搜索结果 |
| 知识库服务 | 项目相关知识 |

**配置格式**：

```json
{
  "mcp_servers": [
    {
      "name": "doc-server",
      "transport": "stdio",
      "command": "doc-mcp-server",
      "args": ["--docs-path", "/path/to/docs"],
      "enabled": true
    },
    {
      "name": "remote-search",
      "transport": "http",
      "url": "https://search.example.com/mcp",
      "auth": {
        "type": "bearer",
        "token_ref": "env:MCP_SEARCH_TOKEN"
      },
      "enabled": true
    }
  ]
}
```

**验收标准**：

- [ ] 外部 MCP 客户端（如 Claude Desktop）能成功连接到平台 MCP 服务
- [ ] 外部客户端能通过 MCP 查询项目符号和调用关系
- [ ] 平台能连接外部 MCP 服务获取额外上下文
- [ ] MCP 通信遵循协议规范

| 功能 | 说明 |
|------|------|

### UC7.7 AI 配置

| 项目 | 内容 |
|------|------|
| **优先级** | P0 |

**配置项**：

| 配置项 | 类型 | 说明 |
|--------|------|------|
| `ai.provider` | Enum | AI 提供者：`openai` / `anthropic` / `ollama` / `custom` |
| `ai.api_key` | String（加密） | API 密钥 |
| `ai.api_base_url` | URL | API 基础 URL（自定义/代理） |
| `ai.model` | String | 模型名称（如 `gpt-4`、`claude-3-opus`、`llama3`） |
| `ai.temperature` | Float | 温度参数（0.0 ~ 2.0），默认 0.3 |
| `ai.max_tokens` | Integer | 最大输出 token 数，默认 4096 |
| `ai.context_window` | Integer | 上下文窗口大小 |
| `ai.system_prompt` | String | 系统提示词（可自定义） |
| `ai.enable_tools` | Boolean | 是否启用技能调用，默认 true |
| `ai.max_tool_calls` | Integer | 单次对话最大技能调用次数，默认 10 |
| `ai.privacy_mode` | Enum | 隐私模式：`full` / `anonymized` / `local_only` |

**AI 提供者支持矩阵**：

| 提供者 | 技能调用 | 流式输出 | 本地运行 | 说明 |
|--------|---------|---------|---------|------|
| OpenAI API | ✅ | ✅ | ❌ | GPT-4、GPT-3.5 |
| Anthropic API | ✅ | ✅ | ❌ | Claude 系列 |
| Ollama | ✅（部分模型） | ✅ | ✅ | 本地开源模型 |
| 自定义 OpenAI 兼容 | ✅ | ✅ | 视情况 | 如 vLLM、LM Studio |

**隐私模式说明**：

| 模式 | 说明 |
|------|------|
| `full` | 完整发送代码片段和项目信息（获得最佳回答质量） |
| `anonymized` | 将代码中的标识符替换为占位符后发送 |
| `local_only` | 仅使用本地模型，不发送任何数据到外部 |

**验收标准**：

- [ ] 支持至少 2 种 AI 提供者（OpenAI + 本地模型）
- [ ] API Key 加密存储
- [ ] 隐私模式按配置正确工作
- [ ] 配置变更后即时生效（无需重启）

## 3. AI 聊天界面设计

### 3.1 布局

```
┌──────────────────────────────────────────┐
│ AI 助手                        [⚙] [📋] │  ← 设置/历史
├──────────────────────────────────────────┤
│ 🔍 当前上下文: src/handler.cpp:42        │  ← 上下文指示
│    函数: process_request                 │
├──────────────────────────────────────────┤
│                                          │
│ 👤 main 函数调用了哪些函数？              │
│                                          │
│ 🤖 让我查询一下 main 函数的调用关系...    │
│    ⏳ 正在调用: get_callees("main")      │
│                                          │
│    main 函数调用了以下 5 个函数：          │
│                                          │
│    1. `init_system()` - src/init.c:15    │
│    2. `load_config()` - src/config.c:28  │
│    3. `start_server()` - src/server.c:42 │
│    4. `run_event_loop()` - src/event.c:7 │
│    5. `cleanup()` - src/main.c:120       │
│                                          │
│    其中 start_server() 和 run_event_loop │
│    是最关键的函数，它们...                 │
│                                          │
├──────────────────────────────────────────┤
│ [📎] 输入你的问题...           [发送 ↵]  │
└──────────────────────────────────────────┘
```

### 3.2 交互细节

- 回答中的代码引用（如 `src/init.c:15`）可点击跳转到编辑器。
- 技能调用过程显示加载动画和技能名称。
- 支持 Markdown 渲染（代码块、列表、表格、链接）。
- 支持代码块的语法高亮。
- 长回答支持折叠/展开。
- 支持复制整个回答或代码块。
- 支持对回答进行评价（👍/👎），用于反馈优化。

## 5. 错误码定义

| 错误码 | 说明 |
|--------|------|
| `E_AI_NOT_CONFIGURED` | AI 提供者未配置 |
| `E_AI_AUTH_FAILED` | API 认证失败 |
| `E_AI_RATE_LIMIT` | API 请求频率超限 |
| `E_AI_CONTEXT_OVERFLOW` | 上下文超出模型窗口限制 |
| `E_AI_MODEL_UNAVAILABLE` | 模型不可用 |
