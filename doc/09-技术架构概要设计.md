# 09 — 技术架构概要设计

> 本文档基于需求规范 v1.2 和各模块细化需求，概述系统整体技术架构。  
> 详细设计将在开发阶段进一步细化。

---

## 1. 架构总览

### 1.1 技术栈

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| **框架** | Tauri 2.x | Rust 后端 + Web 前端的桌面应用框架 |
| **后端语言** | Rust | 高性能、内存安全、并发友好 |
| **前端框架** | Vue 3 + TypeScript | 组合式 API，强类型，生态丰富 |
| **前端构建** | Vite | 快速开发和构建 |
| **UI 组件库** | 待定（如 Naive UI / Element Plus） | 提供基础 UI 组件 |
| **代码编辑器** | Monaco Editor | VS Code 同源，C/C++ 语法支持 |
| **图渲染** | PixiJS + dagre | WebGL 2D 渲染 + 有向图布局 |
| **解析引擎** | libclang (via clang-sys) | C/C++ 语义分析 |
| **辅助解析** | tree-sitter | 快速语法解析（降级方案） |
| **存储** | SQLite (via rusqlite) | 嵌入式关系数据库 |
| **日志** | tracing + tracing-subscriber | 结构化日志框架 |
| **Git 操作** | git2 (libgit2) | Git 仓库操作 |
| **AI 集成** | async-openai / 自定义 HTTP | LLM API 调用 |
| **MCP** | 自研或社区 SDK | Model Context Protocol 实现 |
| **国际化** | vue-i18n | 前端国际化 |
| **状态管理** | Pinia | Vue 3 状态管理 |

## 3. 关键设计决策

### 3.1 前后端通信

**选择 Tauri IPC 而非 REST API**：

- Tauri IPC 是进程内通信，零网络开销
- 支持 Rust 类型自动序列化（serde）
- 原生支持事件和流式传输
- 类型安全（编译时检查）

### 3.2 解析引擎

**选择 libclang 而非自研解析器**：

- libclang 是工业级 C/C++ 解析器，准确度最高
- 完整支持 C/C++ 语义分析（类型推断、重载解析等）
- 通过 Rust FFI（`clang-sys` crate）调用
- 劣势：需要编译数据库获得最佳效果（tree-sitter 作为降级）

### 3.3 存储引擎

**选择 SQLite 而非自定义格式**：

- 成熟稳定，事务支持好
- SQL 查询表达力强，适合关系数据
- 单文件存储，便于备份和迁移
- 读性能优秀（适合查询密集场景）
- WAL 模式支持读写并发

### 3.4 图渲染

**选择 WebGL (PixiJS) 而非 SVG / Canvas 2D**：

- WebGL 硬件加速，大规模节点渲染性能好
- PixiJS 提供场景图管理和事件系统
- 支持高 DPI 渲染
- 劣势：文本渲染需要额外处理（Bitmap Font / MSDF）

## 6. 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| libclang 解析超大项目内存不足 | 解析失败 | 分批解析、内存监控、PCH 缓存 |
| WebGL 图渲染大规模节点卡顿 | 用户体验差 | 聚合策略、LOD、视口裁剪 |
| AI 云端 API 延迟高 | 回答慢 | 支持本地模型、流式输出 |
| 跨平台兼容性问题 | 部分功能失效 | CI 多平台测试、条件编译 |
| 编译数据库缺失 | 解析精度下降 | tree-sitter 降级、引导用户生成 |
