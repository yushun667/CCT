# 06 — 专用分析功能模块细化需求

> 对应需求规范：§3.6 专用分析功能（FR6.1 ~ FR6.3）  
> 优先级范围：P1 ~ P2

---

## 1. 模块概述

专用分析功能模块针对特定的操作系统级项目（Linux Kernel、OpenHarmony）提供深度分析能力，识别其特有的 IPC 通信机制、系统调用接口和内核模式，帮助内核/平台开发者理解底层调用路径。同时提供可扩展的自定义规则机制。

### 1.1 职责边界

- 识别 Linux 内核系统调用（`SYSCALL_DEFINE` 宏族）
- 识别 Linux 内核 ioctl 处理函数
- 识别 OpenHarmony Binder IPC 通信链路
- 识别 OpenHarmony 软总线通信
- 提供自定义 IPC 识别规则的配置和应用

### 1.2 与其他模块的关系

| 上游模块 | 提供内容 |
|----------|----------|
| 源码解析与索引 | AST 节点、符号数据、宏信息 |

| 下游模块 | 消费内容 |
|----------|----------|
| 可视化分析 | IPC/内核路径图数据 |
| 查询与检索 | 系统调用/服务查询 |
| AI 智能助手 | 内核/IPC 领域知识 |

## 2. Linux 内核分析（FR6.1）

### 2.1 系统调用识别

**优先级**：P1

#### 2.1.1 识别规则

Linux 内核使用 `SYSCALL_DEFINEn` 宏定义系统调用（n = 0~6 表示参数个数）：

```c
SYSCALL_DEFINE3(read, unsigned int, fd, char __user *, buf, size_t, count)
{
    // 实现
}
```

**识别策略**：

1. 在 AST 中搜索宏调用节点，匹配 `SYSCALL_DEFINE[0-6]` 模式。
2. 提取宏参数：
   - 第 1 个参数：系统调用名（如 `read`）
   - 后续参数：交替的参数类型和参数名
3. 生成系统调用函数的标准名：`sys_<name>`（如 `sys_read`）。
4. 建立系统调用号与函数的映射关系（可选，需解析系统调用表）。

#### 2.1.2 数据模型

```
SyscallInfo {
  name: String                    // 系统调用名（如 "read"）
  function_name: String           // 内核函数名（如 "sys_read"）
  syscall_number: Option<u32>     // 系统调用号
  param_count: u32                // 参数个数
  parameters: Vec<SyscallParam>   // 参数列表
  file_path: String               // 定义文件
  line: u32                       // 定义行号
  category: SyscallCategory       // 分类（文件IO/进程/内存/网络/IPC...）
}

SyscallParam {
  type_name: String               // 参数类型
  param_name: String              // 参数名
  is_user_pointer: bool           // 是否用户空间指针（__user 标注）
}
```

#### 2.1.3 系统调用表解析

- 解析 `arch/x86/entry/syscalls/syscall_64.tbl`（x86_64）或对应架构的系统调用表文件。
- 格式：`<number> <abi> <name> <entry_point>`
- 建立系统调用号 → 函数名的映射。

**验收标准**：

- [ ] 正确识别 `SYSCALL_DEFINE0` ~ `SYSCALL_DEFINE6` 的所有变体
- [ ] 提取的参数类型和名称正确
- [ ] 能建立系统调用号到函数的映射（当系统调用表可用时）

### 2.2 ioctl 命令识别

**优先级**：P1

#### 2.2.1 识别策略

Linux 驱动通过 `file_operations` 结构体注册 ioctl 处理函数：

```c
static const struct file_operations my_fops = {
    .owner = THIS_MODULE,
    .open = my_open,
    .release = my_release,
    .unlocked_ioctl = my_ioctl,    // ← 目标
    .compat_ioctl = my_compat_ioctl,
};
```

**识别流程**：

1. 在索引中搜索类型为 `struct file_operations` 的全局变量初始化。
2. 提取 `.unlocked_ioctl` 和 `.compat_ioctl` 字段的函数指针值。
3. 追踪 ioctl 处理函数中的 `switch(cmd)` 语句，提取 case 值作为命令码。
4. 解析命令码宏定义（`_IO`、`_IOR`、`_IOW`、`_IOWR`）获取命令详细信息。

#### 2.2.2 命令码解析

```c
#define MY_CMD_GET_INFO  _IOR('M', 1, struct my_info)
#define MY_CMD_SET_MODE  _IOW('M', 2, int)
```

解析为：

```
IoctlCommand {
  name: String                    // 宏名（如 "MY_CMD_GET_INFO"）
  magic: char                     // 幻数（如 'M'）
  number: u32                     // 序号（如 1）
  direction: IoctlDir             // None | Read | Write | ReadWrite
  data_type: String               // 数据类型（如 "struct my_info"）
  handler_function: SymbolId      // 处理函数
  file_path: String               // 宏定义文件
  line: u32
}
```

**验收标准**：

- [ ] 正确识别 `file_operations` 中的 ioctl 处理函数
- [ ] 正确解析 `_IO`/`_IOR`/`_IOW`/`_IOWR` 宏定义的命令码
- [ ] 建立设备文件 → ioctl 命令 → 处理函数的关联

### 2.3 内核调用链追踪

**优先级**：P1

从系统调用入口追踪到具体的驱动/子系统函数：

**追踪策略**：

1. 以系统调用函数为起点（如 `sys_read`）。
2. 利用调用关系索引，逐层展开被调用函数。
3. 识别关键的分层边界函数（VFS 层、文件系统层、块设备层等）。
4. 标注每个函数所属的内核子系统（可通过文件路径规则推断）。

**内核子系统推断规则**：

| 路径模式 | 子系统 |
|----------|--------|
| `kernel/` | 核心内核 |
| `fs/` | 文件系统 |
| `mm/` | 内存管理 |
| `net/` | 网络 |
| `drivers/` | 设备驱动 |
| `arch/` | 体系结构 |
| `block/` | 块设备 |
| `ipc/` | IPC |
| `security/` | 安全 |

## 3. OpenHarmony 分析（FR6.2）

### 3.1 Binder IPC 分析

**优先级**：P1

#### 3.1.1 服务端接口识别

OpenHarmony 使用 IPC 框架，服务端需继承 `IRemoteBroker`：

```cpp
class IMyService : public OHOS::IPC::IRemoteBroker {
public:
    DECLARE_INTERFACE_DESCRIPTOR(u"com.example.IMyService");
    virtual int GetData(int id) = 0;
    virtual void SetMode(int mode) = 0;
};
```

**识别策略**：

1. 在类继承关系中搜索直接或间接继承 `IRemoteBroker` 的类。
2. 提取接口描述符（`DECLARE_INTERFACE_DESCRIPTOR` 宏参数）。
3. 提取接口声明的所有纯虚方法。

#### 3.1.2 Stub（服务端实现）识别

```cpp
class MyServiceStub : public OHOS::IPCObjectStub<IMyService> {
public:
    int OnRemoteRequest(uint32_t code, MessageParcel& data,
                        MessageParcel& reply, MessageOption& option) override;
};
```

**识别策略**：

1. 搜索继承 `IPCObjectStub` 模板类的类。
2. 提取 `OnRemoteRequest` 方法的实现。
3. 分析 `switch(code)` 语句，提取消息码与处理逻辑的映射。

#### 3.1.3 Proxy（客户端代理）识别

```cpp
class MyServiceProxy : public OHOS::IPCObjectProxy<IMyService> {
public:
    int GetData(int id) override {
        MessageParcel data, reply;
        data.WriteInt32(id);
        Remote()->SendRequest(CMD_GET_DATA, data, reply, option);
        return reply.ReadInt32();
    }
};
```

**识别策略**：

1. 搜索继承 `IPCObjectProxy` 模板类的类。
2. 提取 `SendRequest` / `Transact` 调用，获取消息码。
3. 建立 Proxy 方法 → 消息码 → Stub 处理 的映射。

#### 3.1.4 IPC 通信模型

```
IPCService {
  interface_name: String          // 接口名（如 "IMyService"）
  descriptor: String              // 接口描述符
  stub_class: SymbolId            // Stub 实现类
  proxy_class: SymbolId           // Proxy 代理类
  methods: Vec<IPCMethod>         // IPC 方法列表
}

IPCMethod {
  name: String                    // 方法名
  code: u32                       // 消息码
  proxy_function: SymbolId        // Proxy 端函数
  stub_handler: SymbolId          // Stub 端处理函数
  parameters: Vec<ParamInfo>      // 参数列表
  return_type: String             // 返回类型
}
```

### 3.2 服务注册与发现

**优先级**：P1

**识别目标**：

- 服务注册：`SystemAbilityManagerClient::GetInstance()->AddSystemAbility(SA_ID, service)`
- 服务获取：`SystemAbilityManagerClient::GetInstance()->GetSystemAbility(SA_ID)`

**识别策略**：

1. 搜索 `AddSystemAbility` / `AddService` 调用。
2. 提取 SA_ID（System Ability ID）参数值。
3. 提取注册的服务对象。
4. 搜索 `GetSystemAbility` / `GetService` 调用。
5. 建立 SA_ID → 服务接口 → 客户端使用点的映射。

```
ServiceRegistration {
  sa_id: u32                      // System Ability ID
  service_interface: String       // 服务接口名
  registration_site: Location     // 注册位置
  stub_class: SymbolId            // 服务实现类
}

ServiceDiscovery {
  sa_id: u32
  discovery_site: Location        // 获取位置
  consumer_function: SymbolId     // 消费者函数
}
```

### 3.3 软总线分析

**优先级**：P2

| API 模式 | 说明 |

## 4. 自定义规则（FR6.3）

**优先级**：P2

### 4.1 规则定义格式

用户可通过 YAML 配置文件定义自定义 IPC 识别规则：

```yaml
rules:
  - name: "Custom RPC Framework"
    description: "识别自定义 RPC 框架的服务注册和调用"
    version: "1.0"
    
    patterns:
      # 服务接口识别
      - type: "service_interface"
        match:
          kind: "class_inheritance"
          base_class: "MyRPCService"
        extract:
          - field: "interface_name"
            from: "class_name"
          - field: "methods"
            from: "virtual_methods"
      
      # 服务注册识别
      - type: "service_registration"
        match:
          kind: "function_call"
          function: "RPCRegistry::Register"
        extract:
          - field: "service_name"
            from: "argument[0]"
          - field: "service_instance"
            from: "argument[1]"
      
      # 客户端调用识别
      - type: "client_call"
        match:
          kind: "function_call"
          function: "RPCClient::Call"
        extract:
          - field: "target_service"
            from: "argument[0]"
          - field: "method_name"
            from: "argument[1]"
    
    relations:
      - type: "ipc_call"
        from: "client_call.caller_function"
        to: "service_interface.methods[method_name]"
        via: "service_registration"
```

### 4.2 规则匹配方式

| 匹配方式 | 说明 | 适用场景 |
|----------|------|----------|
| **类继承匹配** | 匹配继承特定基类的类 | 服务接口识别 |
| **函数调用匹配** | 匹配对特定函数的调用 | 服务注册/调用识别 |
| **宏调用匹配** | 匹配特定宏的使用 | 系统调用、接口声明 |
| **结构体字段匹配** | 匹配结构体特定字段的赋值 | 回调注册（如 file_operations） |
| **正则表达式匹配** | 对符号名/代码文本的正则匹配 | 通用模式 |

### 4.3 规则管理

- 规则文件存储在项目配置目录：`<project>/rules/*.yaml`
- 支持全局规则目录：`<app_data>/rules/`（对所有项目生效）
- 内置规则（Linux Kernel、OpenHarmony）不可修改，但可禁用
- 用户规则可创建、编辑、启用/禁用、删除
- 提供规则验证工具：检查规则语法和匹配效果

**验收标准**：

- [ ] 自定义规则能正确匹配目标代码模式
- [ ] 规则定义语法校验通过后才能保存
- [ ] 规则匹配结果集成到可视化和查询模块
- [ ] 内置规则覆盖 Linux Kernel 主要系统调用和 OpenHarmony Binder IPC

## 6. 错误码定义

| 错误码 | 说明 |
|--------|------|
| `E_KERNEL_NO_SYSCALL` | 未检测到系统调用定义（可能不是内核项目） |
| `E_KERNEL_TABLE_PARSE` | 系统调用表解析失败 |
| `E_OH_NO_IPC` | 未检测到 IPC 接口定义 |
| `E_OH_INCOMPLETE_IPC` | IPC 链路不完整（缺少 Stub 或 Proxy） |
| `E_RULE_SYNTAX` | 自定义规则语法错误 |
| `E_RULE_NO_MATCH` | 自定义规则未匹配到任何结果 |
