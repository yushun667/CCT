## 1. 性能需求（NFR1.x）

### NFR1.1 解析速度

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 中型项目（10 万行） | < 2 分钟 | 全量解析计时 |
| 大型项目（100 万行） | < 30 分钟 | 全量解析计时 |
| 超大项目（1000 万行，如 Linux Kernel） | < 120 分钟 | 全量解析计时 |
| 增量解析（单文件变更） | < 5 秒 | 增量解析计时 |
| 增量解析（100 文件变更） | < 60 秒 | 增量解析计时 |

**优化措施**：

| 措施 | 预期效果 | 优先级 |
|------|----------|--------|
| 并行解析（多线程） | 线性加速比 0.7N（N = 核心数） | P0 |
| PCH 预编译头缓存 | 头文件重复解析降低 80% | P0 |
| 增量解析 | 变更场景解析时间降低 95% | P0 |
| AST 缓存 | 重启后避免重新解析 | P1 |
| 内存映射文件 I/O | 大文件读取速度提升 30% | P1 |

**测试环境基准**：

- CPU：8 核 16 线程（如 Apple M1 Pro 或同级）
- 内存：16 GB
- 存储：SSD
- 测试项目：Linux Kernel 6.x 源码（约 2800 万行，包含注释和空行）

### NFR1.2 查询响应

| 查询类型 | 目标响应时间 | 说明 |
|----------|-------------|------|
| 符号搜索（模糊） | < 100ms | 返回前 50 条结果 |
| 调用查询（直接） | < 200ms | depth=1 |
| 调用查询（递归） | < 2s | depth≤5 |
| 引用查询 | < 500ms | 全量搜索 |
| 路径查询 | < 5s | 最短路径 |
| 包含查询 | < 200ms | 直接包含 |

**优化措施**：

| 措施 | 说明 |
|------|------|
| 符号名 Trie 索引 | 支持前缀/模糊搜索的快速匹配 |
| 调用关系邻接表（内存） | 避免每次查询都访问磁盘 |
| 查询结果缓存（LRU） | 相同查询直接返回缓存 |
| 异步查询 + 取消机制 | 用户输入变化时取消旧查询 |

### NFR1.3 图渲染

| 指标 | 目标值 |
|------|--------|
| 节点数 < 500 | 首次渲染 < 1s，交互 60fps |
| 节点数 500~5000 | 首次渲染 < 3s，交互 30fps |
| 节点数 > 5000 | 使用聚合/懒加载，交互 30fps |
| 布局计算（层次布局，1000 节点） | < 2s |
| 布局计算（力导向，1000 节点） | < 5s（含迭代稳定） |

### NFR1.4 内存占用

| 场景 | 最大内存 | 说明 |
|------|----------|------|
| 空闲状态 | < 200MB | 无项目打开 |
| 中型项目已加载索引 | < 1GB | 10 万行项目 |
| 大型项目已加载索引 | < 4GB | 100 万行项目 |
| 超大项目解析中 | < 8GB | 解析过程峰值 |
| 图渲染（5000 节点） | + 200MB | 增量内存占用 |

**内存管理策略**：

- 解析完成后释放 AST 对象，仅保留索引数据
- 索引数据按需加载到内存（冷数据保留在磁盘）
- 图渲染使用对象池，减少 GC 压力
- 内存使用监控，接近阈值时提示用户

### NFR1.5 远程解析与传输性能

| 指标 | 目标值 | 说明 |
|------|--------|------|
| SSH 连接建立 | < 5 秒 | 含认证（密钥认证） |
| Agent 启动响应 | < 3 秒 | Agent 接收首条 RPC |
| 远程全量解析（100 万行，8 核服务器） | < 30 分钟 | 利用远程 CPU |
| 索引数据传输（100MB 索引，100Mbps） | < 30 秒 | zstd 压缩后约 30-40MB |
| 增量索引传输（100 文件变更） | < 10 秒 | 仅传输变更部分 |
| 远程文件浏览（目录列表） | < 2 秒 | SFTP 目录读取 |
| 远程文件读取（< 1MB） | < 3 秒 | 编辑器打开远程文件 |

**优化措施**：

| 措施 | 说明 |
|------|------|
| SSH ControlMaster | 复用 SSH 连接，避免重复认证和 TCP 握手 |
| SSH Keep-Alive | 定期心跳防止连接被防火墙断开 |
| zstd 压缩 | 索引数据压缩率 60-70%，减少传输量 |
| 增量传输 | 增量解析后仅传输差异数据 |
| 目录缓存 | SFTP 目录列表结果缓存 60 秒 |
| 文件内容缓存 | 已读取的远程文件内容 LRU 缓存 |

## 2. 可用性需求（NFR2.x）

### NFR2.1 界面设计原则

| 原则 | 说明 |
|------|------|
| **一致性** | 全应用使用统一的控件样式、图标风格、颜色方案 |
| **可发现性** | 常用功能不超过 3 次点击可达 |
| **反馈及时** | 所有操作 200ms 内给出视觉反馈（加载指示器） |
| **容错性** | 可撤销的操作提供撤销机制；不可逆操作二次确认 |
| **可定制** | 支持暗色/亮色主题、面板布局可拖拽调整 |

### NFR2.3 错误提示设计

| 错误级别 | 显示方式 | 持续时间 |
|----------|----------|----------|
| **信息** | 底部状态栏蓝色消息 / Toast 通知 | 5 秒自动消失 |
| **警告** | 底部状态栏黄色消息 / Toast 通知 | 需用户关闭 |
| **错误** | 对话框 / Toast 通知 + 详情链接 | 需用户确认 |

每条错误提示包含：

1. 简短描述（用户可理解的语言）
2. 可能的原因列表
3. 建议的解决方案
4. "查看详情"链接（展开技术细节/日志）
5. "复制错误信息"按钮（便于反馈）

| 要求 | 说明 |
|------|------|

- 支持 SSH 密钥文件选择和连接测试
- 支持 SSH Host Key 指纹管理（查看、信任、删除）

## 4. 可靠性需求（NFR4.x）

### NFR4.1 数据持久性

| 数据类型 | 存储方式 | 恢复策略 |
|----------|----------|----------|
| 项目配置 | JSON 文件 | 启动时自动加载 |
| 解析索引 | SQLite 数据库 | 启动时加载，损坏时提示重建 |
| 对话历史 | JSON 文件 | 启动时按需加载 |
| 用户设置 | JSON 文件 | 启动时加载，损坏时回退默认 |
| 认证信息 | 系统密钥链 / 加密文件 | 启动时按需读取 |

**数据目录结构**：

```
<app_data>/
├── config.json                    // 全局配置
├── projects/
│   ├── <project_id>/
│   │   ├── project.json           // 项目配置
│   │   ├── index.db               // 解析索引（SQLite）
│   │   ├── auth.enc               // 加密认证信息
│   │   ├── conversations/         // 对话历史
│   │   │   ├── <conv_id>.json
│   │   │   └── ...
│   │   └── rules/                 // 自定义规则
│   │       └── *.yaml
│   └── ...
├── rules/                         // 全局自定义规则
│   └── *.yaml
└── logs/                          // 日志文件
    ├── app.log
    └── app.log.1
```

### NFR4.2 崩溃恢复

| 崩溃场景 | 恢复策略 |
|----------|----------|
| 本地解析过程中崩溃 | 已完成文件的索引保留，重启后提示"上次解析未完成，是否继续？" |
| 远程解析中桌面端崩溃 | 远程 Agent 继续完成解析，桌面端重启后可查询 Agent 状态并获取结果 |
| 远程解析中 Agent 崩溃 | 桌面端提示"远程 Agent 异常退出"，可重新启动 Agent 并重试解析 |
| SSH 连接断开 | 自动重连（可配置重试策略），若 Agent 在运行则解析不中断 |
| 图渲染崩溃 | 不影响索引数据，重新打开图视图即可 |
| AI 对话中崩溃 | 已保存的对话历史不丢失，重启后可继续 |

**崩溃报告**：

- 未捕获异常自动记录到日志
- 生成崩溃报告文件（包含堆栈跟踪、系统信息、最近操作）
- 重启后提示"上次异常退出，是否发送崩溃报告？"（可选）

### NFR4.3 数据一致性

- SQLite 使用 WAL 模式，保证写入原子性
- 增量解析使用事务：先删除旧数据，再插入新数据，事务失败自动回滚
- 并行解析完成后执行引用完整性检查
- 定期（可配置）执行索引一致性校验

### NFR4.4 网络容错

| 场景 | 处理策略 |
|------|----------|
| SSH 连接超时 | 超时时间可配置（默认 15s），超时后提示用户 |
| SSH 连接断开 | 自动重连，最多重试 3 次，间隔 2s/5s/10s |
| SSH Keep-Alive 超时 | 重新建立连接，检查 Agent 状态 |
| 索引传输中断 | 支持断点续传（记录已传输字节偏移） |
| Agent 无响应 | 超时 30s 后判定无响应，提示用户重启 Agent |
| 跳板机断开 | 尝试通过跳板机重新建立连接 |
| 远程服务器重启 | 检测到连接断开后提示用户，可手动重连并重新启动 Agent |

### NFR4.5 认证安全

| 平台 | 加密方案 |
|------|----------|
| macOS | Keychain Services API |
| Windows | Windows Credential Manager (DPAPI) |
| Linux | libsecret (GNOME Keyring / KDE Wallet) |
| 降级方案 | AES-256-GCM 加密，密钥派生自硬件指纹 |

## 5. 安全性需求（NFR5.x）

### NFR5.1 本地数据安全

- 所有解析数据、索引、项目配置仅存储在用户本地
- 不收集用户使用数据（除非用户主动开启匿名统计）
- 应用更新检查仅发送版本号，不发送其他信息

### NFR5.2 AI 隐私

- 使用云端 AI 时，首次使用弹出隐私提示对话框
- 提示内容明确说明哪些数据会发送到外部服务
- 提供三级隐私模式（见 AI 模块文档 §UC7.7）
- 匿名化模式下，替换标识符（函数名、变量名）为占位符
- 日志中不记录发送到 AI 的代码内容

### NFR5.3 远程访问安全

- 所有远程通信强制通过 SSH 加密通道
- SSH 密钥密码短语存储在操作系统密钥链中
- 不支持 Telnet 等明文协议
- 首次连接时验证远程主机指纹，存储到本地 known_hosts
- 主机指纹变化时警告用户（防中间人攻击）
- Agent 仅监听 SSH 通道（stdin/stdout），不开放网络端口
- Agent 以当前用户权限运行，不要求 root 权限

## 6. 可扩展性需求（NFR6.x）

### NFR6.1 插件机制

**插件类型**：

| 插件类型 | 扩展能力 | 优先级 |
|----------|----------|--------|
| 分析规则插件 | 新增 IPC 识别规则、代码模式检测 | P1 |
| 可视化插件 | 自定义图布局算法、节点渲染样式 | P2 |
| AI 技能插件 | 新增 AI 可调用的技能 | P1 |
| 导出格式插件 | 新增图导出格式 | P2 |
| 主题插件 | 自定义编辑器和 UI 主题 | P2 |

**插件接口设计原则**：

- 插件通过明确定义的接口（Trait / Interface）与主应用交互
- 插件运行在沙箱环境中，不可访问主应用内部状态
- 插件使用声明式配置文件描述元信息（名称、版本、依赖、权限）
- 插件可通过应用内插件管理器安装/卸载/启用/禁用

### NFR6.2 开放 API

- 提供 CLI 工具，支持命令行调用解析和查询功能
- 提供 HTTP API（可选启动），供外部脚本和工具调用
- API 遵循 RESTful 设计规范
- 提供 API 文档（OpenAPI/Swagger 格式）

**CLI 示例**：

```bash
# 解析项目
cct parse --project /path/to/source --compile-db /path/to/compile_commands.json

# 查询符号
cct query symbols --name "process_request" --kind function

# 查询调用者
cct query callers --function "sys_read" --depth 3

# 查询调用路径
cct query path --from "sys_read" --to "ext4_file_read"

# 导出调用图
cct export call-graph --root "main" --depth 5 --format svg --output graph.svg
```

## 7. 日志系统需求

### 7.1 日志级别

| 级别 | 使用场景 |
|------|----------|
| `TRACE` | 详细的调试信息（函数进入/退出、变量值） |
| `DEBUG` | 调试信息（处理步骤、中间结果） |
| `INFO` | 运行信息（操作开始/完成、配置加载） |
| `WARN` | 警告信息（非致命异常、降级处理） |
| `ERROR` | 错误信息（操作失败、异常捕获） |
| `FATAL` | 致命错误（需要退出应用） |

### 7.2 日志输出

| 输出目标 | 说明 |
|----------|------|
| 控制台 | 开发模式下输出到 stderr |
| 文件 | 日志文件落盘，支持滚动（按大小或日期） |
| UI | ERROR 级别以上的日志在 UI 通知区域显示 |

### 7.3 日志文件管理

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 最大单文件大小 | 10MB | 超出后滚动到新文件 |
| 最大日志文件数 | 10 | 超出后删除最旧的 |
| 日志保留天数 | 30 天 | 超过后自动清理 |
| 默认日志级别 | INFO | 可通过配置调整 |

### 7.4 日志格式

```
[2026-03-01T14:30:15.123+08:00] [INFO] [parse::engine] 开始解析文件: src/main.c (1/500)
[2026-03-01T14:30:15.456+08:00] [DEBUG] [parse::symbol] 提取函数符号: main (src/main.c:10)
[2026-03-01T14:30:16.789+08:00] [ERROR] [parse::engine] 解析失败: src/broken.c - 语法错误 (line 42)
```

格式：`[时间戳] [级别] [模块] 消息`

### 7.5 日志要求

- 每个公开函数在入口处记录 `TRACE` 级别日志（函数名和关键参数）
- 每个后端命令（Tauri Command）在入口记录 `INFO` 级别日志
- 所有异常和错误捕获记录 `ERROR` 级别日志，包含上下文信息
- 性能关键路径记录耗时信息（`DEBUG` 级别）
- 敏感信息（密码、token）禁止出现在任何级别的日志中
