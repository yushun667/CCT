# 04 — 可视化分析模块细化需求

> 对应需求规范：§3.4 可视化分析（FR4.1 ~ FR4.5）  
> 优先级范围：P0 ~ P2

---

## 1. 模块概述

可视化分析模块是平台的核心交互界面，将索引数据转化为直观的图形化视图，帮助用户理解代码架构、模块关系和调用路径。支持多种图类型和丰富的交互操作。

### 1.1 职责边界

- 渲染函数级调用图
- 渲染模块级依赖图
- 渲染文件级依赖图
- 渲染 IPC / 内核调用路径图
- 提供图交互操作（缩放、平移、拖拽、选择、导出）
- 提供与代码编辑器的双向联动

### 1.2 与其他模块的关系

| 上游模块 | 提供内容 |
|----------|----------|
| 查询与检索 | 图数据（节点/边列表） |
| 项目管理 | 模块定义（用于聚合） |

| 下游模块 | 交互方式 |
|----------|----------|
| 代码编辑器 | 节点点击 → 跳转源码 |
| AI 智能助手 | 图上下文 → AI 感知 |

## 2. 图类型详细描述

### 2.1 调用图（FR4.1）

**优先级**：P0

#### 2.1.1 数据模型

```
CallGraphNode {
  id: SymbolId
  label: String                   // 显示名称（可配置：短名/全名）
  qualified_name: String
  file_path: String
  line: u32
  node_type: NodeType             // Normal | SystemCall | Callback | Virtual | External
  module: Option<String>          // 所属模块名
  metrics: NodeMetrics            // 度量信息
  position: Option<Position>      // 布局位置（由布局算法计算）
  collapsed: bool                 // 是否折叠子图
  visible: bool                   // 是否可见
}

NodeMetrics {
  in_degree: u32                  // 入度（被调用次数）
  out_degree: u32                 // 出度（调用次数）
  complexity: Option<u32>         // 圈复杂度
  line_count: Option<u32>         // 函数体行数
}

CallGraphEdge {
  source_id: SymbolId             // 调用者
  target_id: SymbolId             // 被调用者
  edge_type: EdgeType             // Direct | Virtual | Indirect | Callback
  weight: u32                     // 调用次数
  call_sites: Vec<Location>       // 调用位置列表
}
```

#### 2.1.2 节点着色方案

| 节点类型 | 颜色 | 说明 |
|----------|------|------|
| 普通函数 | `#4A90D9`（蓝色） | 默认函数节点 |
| 系统调用 | `#E74C3C`（红色） | SYSCALL_DEFINE 定义的函数 |
| 回调函数 | `#F39C12`（橙色） | 通过函数指针注册的回调 |
| 虚函数 | `#9B59B6`（紫色） | 虚函数/虚方法 |
| 外部函数 | `#95A5A6`（灰色） | 未在项目中定义的函数 |
| 高亮节点 | `#2ECC71`（绿色） | 当前选中或搜索命中 |
| 路径节点 | `#E67E22`（深橙色） | 路径查询结果中的节点 |

用户可自定义配色方案。

#### 2.1.3 筛选与过滤

| 过滤器 | 说明 | UI 控件 |
|--------|------|---------|
| 模块过滤 | 仅显示指定模块的函数 | 多选下拉框 |
| 文件过滤 | 仅显示指定文件/目录的函数 | 文件树勾选 |
| 深度过滤 | 限制从焦点节点展开的深度 | 数字输入/滑块 |
| 类型过滤 | 按节点类型筛选 | 复选框组 |
| 度过滤 | 隐藏度数低于阈值的节点 | 滑块 |
| 搜索过滤 | 按名称筛选可见节点 | 搜索框 |

#### 2.1.4 展开/折叠

- **按模块折叠**：将同模块的函数节点折叠为一个模块节点，显示模块名和内部函数数量。
- **按文件折叠**：将同文件的函数折叠为一个文件节点。
- **手动折叠**：右键节点选择"折叠子图"，将其下游所有节点收缩为一个节点。
- **逐步展开**：双击折叠节点可展开一层子节点。

#### 2.1.5 布局算法

| 算法 | 适用场景 | 说明 |
|------|----------|------|
| **层次布局（Hierarchical/Sugiyama）** | 调用层级清晰的图 | 从上到下或从左到右分层排列，最小化交叉边。**默认布局**。 |
| **力导向布局（Force-Directed）** | 节点关系均匀的图 | 模拟物理弹簧模型，自动聚集密切相关节点。 |
| **径向布局（Radial）** | 以某函数为中心的探索 | 焦点节点居中，相关节点按距离呈环形分布。 |
| **正交布局（Orthogonal）** | 方框流程图风格 | 使用直角折线连接，适合较小的图。 |

用户可在布局间切换，切换时有平滑动画过渡。

### 2.2 模块依赖图（FR4.2）

**优先级**：P0

#### 2.2.1 模块定义方式

| 方式 | 说明 | 示例 |
|------|------|------|
| **目录映射** | 将一个目录映射为一个模块 | `/src/net/` → 模块 "网络层" |
| **文件列表** | 手动指定属于模块的文件列表 | `[a.c, b.c, c.h]` → 模块 "认证" |
| **正则表达式** | 用路径正则匹配文件到模块 | `src/driver/.*` → 模块 "驱动" |
| **自动推断** | 基于顶层目录自动创建模块（P2） | 自动按 `src/` 下一级目录分模块 |

#### 2.2.2 依赖计算

```
ModuleDependency {
  source_module: String           // 依赖方模块
  target_module: String           // 被依赖模块
  call_count: u64                 // 跨模块调用次数
  include_count: u64              // 跨模块包含次数
  functions_involved: Vec<(SymbolId, SymbolId)>  // 涉及的函数对
  strength: f32                   // 依赖强度 = log2(call_count + include_count)
}
```

**依赖聚合规则**：

1. 遍历所有调用关系，若调用者和被调用者属于不同模块，记为一次跨模块调用。
2. 遍历所有包含关系，若包含文件和被包含文件属于不同模块，记为一次跨模块包含。
3. 同方向的多次跨模块关系聚合为一条依赖边，权重 = 调用次数 + 包含次数。
4. 边的粗细与权重成正比（使用 `log2` 缩放避免极端值）。

#### 2.2.3 循环依赖检测

- 使用拓扑排序算法检测有向图中的环。
- 检测到的循环依赖以红色虚线高亮显示。
- 悬停在循环边上时，显示涉及的具体函数/文件列表。
- 提供"循环依赖报告"面板，列出所有检测到的循环依赖及建议。

#### 2.2.4 数据模型

```
ModuleGraphNode {
  name: String                    // 模块名
  file_count: u32                 // 包含文件数
  symbol_count: u32               // 包含符号数
  total_lines: u64                // 总代码行数
  color: String                   // 显示颜色
  position: Position
}

ModuleGraphEdge {
  source: String                  // 依赖方
  target: String                  // 被依赖方
  weight: u64                     // 依赖权重
  is_cyclic: bool                 // 是否属于循环依赖
  details: ModuleDependency       // 详细依赖信息
}
```

**验收标准**：

- [ ] 用户能通过三种方式（目录/文件列表/正则）定义模块
- [ ] 依赖图正确反映模块间调用和包含关系
- [ ] 边粗细与依赖强度成正比
- [ ] 循环依赖以红色高亮标识
- [ ] 点击依赖边可查看涉及的具体函数列表

### 2.3 文件依赖图（FR4.3）

**优先级**：P1

#### 2.3.1 数据模型

```
FileGraphNode {
  file_path: String               // 文件路径
  file_name: String               // 文件名（显示用）
  directory: String               // 所在目录
  symbol_count: u32               // 文件中的符号数
  line_count: u32                 // 文件行数
  is_header: bool                 // 是否头文件
}

FileGraphEdge {
  source: String                  // 包含方文件
  target: String                  // 被包含文件
  include_line: u32               // #include 所在行
  is_system: bool                 // 是否系统头文件
}
```

#### 2.3.2 目录聚合

- 支持按目录层级聚合文件节点：
  - 同目录下的文件可折叠为一个"目录节点"
  - 目录节点显示：目录名、文件数、聚合后的入度/出度
- 聚合层级可调（顶层目录 / 二级目录 / 三级目录 ...）

#### 2.3.3 过滤选项

- 排除系统头文件（默认排除）
- 排除指定目录（如 `test/`、`third_party/`）
- 仅显示与指定文件相关的依赖子图

**验收标准**：

- [ ] 正确展示文件间的 `#include` 关系
- [ ] 支持按目录层级聚合显示
- [ ] 可过滤系统头文件
- [ ] 节点数过多时支持懒加载或聚合

### 2.4 IPC/内核调用路径图（FR4.4）

**优先级**：P1

#### 2.4.1 Linux 内核路径图

**入口识别**：

- 扫描 `SYSCALL_DEFINE0` ~ `SYSCALL_DEFINE6` 宏，提取系统调用入口函数。
- 扫描 `ioctl` 处理函数（通过 `file_operations` 结构体的 `.unlocked_ioctl` 字段）。

**路径图结构**：

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌───────────┐
│ 系统调用入口  │ ──→ │ VFS 层函数    │ ──→ │ 文件系统实现  │ ──→ │ 块设备驱动  │
│ sys_read     │     │ vfs_read     │     │ ext4_read    │     │ blk_read  │
└─────────────┘     └──────────────┘     └─────────────┘     └───────────┘
```

- 节点按内核层级着色（系统调用层 → VFS 层 → 文件系统层 → 块设备层 → 硬件抽象层）。
- 支持选择特定系统调用作为起点，展示到底层驱动的完整路径。

#### 2.4.2 OpenHarmony 路径图

**服务调用路径**：

```
┌────────────┐   Transact    ┌──────────────┐   OnRemoteRequest  ┌────────────┐
│ 客户端 Proxy │ ──────────→  │ Binder Driver │ ────────────────→  │ 服务端 Stub │
│ (SA Client) │              │              │                     │ (SA Server) │
└────────────┘              └──────────────┘                     └────────────┘
```

- 展示服务注册 → 服务发现 → RPC 调用的完整链路。
- 节点标注服务名、接口名、方法名。
- 边标注消息码（command code）。

**验收标准**：

- [ ] 能正确识别系统调用入口函数
- [ ] 路径图展示内核分层调用关系
- [ ] OpenHarmony Binder 通信路径正确追踪
- [ ] 支持从入口函数追踪到底层实现

## 3. 性能优化策略

### 3.1 大规模图渲染（NFR1.3）

| 节点数量 | 策略 |
|----------|------|
| < 500 | 直接渲染所有节点，完整交互 |
| 500 ~ 5,000 | 使用 WebGL 渲染（Canvas），简化节点样式 |
| 5,000 ~ 50,000 | 聚合显示 + 语义缩放（缩放到阈值以下时显示聚合节点） |
| > 50,000 | 视口渲染（仅渲染可见区域节点），虚拟滚动 |

### 3.2 语义缩放

- **高缩放级别**（放大）：显示完整节点信息（函数名、参数、类型）
- **中缩放级别**：仅显示函数名
- **低缩放级别**（缩小）：节点缩小为圆点，标签隐藏
- **最低级别**：按模块/文件聚合为大色块

### 3.3 懒加载

- 初始仅加载焦点节点及其直接相邻节点。
- 用户展开节点时动态加载下一层。
- 加载过程中显示加载动画。

## 4. 前端技术选型

| 技术 | 用途 | 说明 |
|------|------|------|
| **Canvas/WebGL** | 图渲染 | 高性能 2D 渲染 |
| **d3-force** | 力导向布局 | D3.js 的力模拟模块 |
| **dagre** | 层次布局 | 有向图分层布局算法 |
| **pixi.js 或 konva.js** | 渲染引擎 | 基于 WebGL 的 2D 渲染框架 |

推荐使用 **pixi.js** + **dagre** 组合，兼顾渲染性能和布局质量。

## 6. 错误码定义

| 错误码 | 说明 |
|--------|------|
| `E_GRAPH_NO_DATA` | 无图数据（索引未建立或无匹配结果） |
| `E_GRAPH_TOO_LARGE` | 图规模超过渲染上限 |
| `E_GRAPH_LAYOUT_FAIL` | 布局算法计算失败 |
| `E_GRAPH_EXPORT_FAIL` | 导出失败 |
| `E_MODULE_NOT_DEFINED` | 未定义模块（模块依赖图需要） |
