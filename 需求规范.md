## C/C++ 架构分析平台需求规范 v1.3（SSH 远程服务器直接解析）

### 1. 引言

#### 1.1 背景
大型 C/C++ 项目（如 OpenHarmony、Linux Kernel）源码规模庞大，模块间关系复杂，传统阅读方式难以快速理解整体架构、模块依赖、关键调用路径。开发者和架构师需要一款专业的分析工具，能够深度解析源码，提取架构信息，并提供可视化的交互界面，辅助代码理解和重构。实际开发中，源码往往存储于远端开发服务器（如企业构建服务器、云开发机），且项目规模巨大（如 Linux Kernel 数十 GB），不适合也不必拉取到本地。因此需要支持通过 SSH 直接登录远程服务器，在远程服务器上就地解析代码，并将索引结果回传到本地应用进行查询和可视化。

#### 1.2 目标
构建一个工程级的桌面应用程序，能够：
- 对大规模 C/C++ 源码进行静态分析，提取符号、调用关系、包含关系等。
- 支持多种维度可视化（调用图、模块依赖图、文件依赖图）。
- 提供与代码视图的双向联动，便于定位和探索。
- 针对操作系统级项目（Linux、OpenHarmony）提供专用的 IPC 和内核调用分析。
- **支持通过 SSH 登录远程服务器，直接访问和浏览远程代码库，在远程服务器上执行解析（通过部署远程 Agent），索引数据回传本地，无需将源码下载到本地。**
- 集成 AI 能力，通过自然语言问答、技能调用、多 Agent 协同等方式，智能辅助架构理解。

#### 1.3 适用范围
本需求规范定义了“C/C++ 架构分析平台”的所有功能和非功能需求，作为产品开发、测试和验收的依据。

### 2. 用户角色

| 角色 | 描述 |
|------|------|
| **架构师** | 负责系统整体设计，需要理解模块划分、依赖关系、关键路径，进行架构评审和重构决策。 |
| **内核/平台开发者** | 专注于操作系统内核或底层模块开发，需要追踪系统调用、驱动调用链、IPC 通信路径。 |
| **普通开发者** | 需要快速了解陌生模块的结构、函数调用关系，定位代码位置。 |
| **技术管理者** | 需要宏观查看项目结构、模块耦合度，辅助技术规划。 |
| **远程团队协作成员** | 需要分析存储在远端服务器上的代码，通过 SSH 连接远程开发机进行代码分析。 |

### 3. 功能需求

#### 3.1 项目管理
- **FR1.1 本地项目创建**：用户可新建本地项目，指定本地源码根目录，配置项目名称。
- **FR1.2 远程项目创建**：用户可新建远程项目，通过 SSH 协议连接远程服务器（提供主机地址、端口、用户名、SSH 密钥或密码），指定远程服务器上的源码根目录路径。应用不将源码下载到本地，而是通过 SSH 直接访问远程文件系统。
- **FR1.3 编译数据库导入**：支持从本地或远程项目导入 `compile_commands.json` 文件。对于远程项目，通过 SSH/SFTP 读取远程服务器上的编译数据库文件；若不存在，可引导用户在远程服务器上生成（通过 SSH 执行远程命令）。
- **FR1.4 项目配置**：支持保存项目配置（包括源码路径、SSH 连接信息、编译数据库、模块定义等），下次启动可快速加载。SSH 认证信息需加密存储。
- **FR1.5 多项目切换**：支持同时管理多个本地和远程项目，在项目间切换。
- **FR1.6 远程 Agent 管理**：支持在远程服务器上部署、启动、停止和升级解析 Agent（`cct-agent`）。Agent 是一个轻量级命令行程序，在远程服务器上执行实际的源码解析工作，并将索引数据回传给本地应用。
- **FR1.7 远程文件浏览**：对于远程项目，通过 SSH/SFTP 浏览远程服务器上的文件目录结构，支持在编辑器中查看远程文件内容。
- **FR1.8 远程操作进度与状态**：在远程连接、Agent 部署、远程解析等过程中，提供实时进度反馈和状态显示。

#### 3.2 源码解析与索引
- **FR2.1 符号提取**：从源码中提取函数、变量、类型（结构体/类/枚举）、宏的定义和声明，记录其名称、完全限定名、文件位置（文件路径、行号、列号）、属性（是否为定义、访问权限等）。
- **FR2.2 调用关系提取**：识别函数调用表达式，记录调用者（caller）和被调用者（callee），支持跨文件调用。
- **FR2.3 包含关系提取**：记录 `#include` 指令，建立文件间的包含关系图。
- **FR2.4 宏处理**：提取宏定义位置，记录宏展开后的引用关系（可选，可先记录宏定义）。
- **FR2.5 条件编译处理**：结合编译数据库中的宏定义，正确解析条件编译块内的符号。
- **FR2.6 增量解析**：当源码文件发生变化时，支持重新解析修改的文件，并更新索引，避免全量重扫。对于远程项目，远程 Agent 可检测文件变更并执行增量解析。
- **FR2.7 解析进度反馈**：在解析过程中，向前端提供进度通知（文件数、百分比等）。
- **FR2.8 并行解析**：支持多线程或多进程并发解析多个源码文件，充分利用多核 CPU 资源，显著提升大型项目的解析速度。解析过程中需保证数据一致性，避免竞态条件。
- **FR2.9 远程解析**：对于远程项目，解析工作由部署在远程服务器上的 Agent（`cct-agent`）执行，利用远程服务器的 CPU 和内存资源。解析完成后，Agent 将索引数据（SQLite 数据库）通过 SSH 通道回传到本地应用。

#### 3.3 查询与检索
- **FR3.1 符号搜索**：支持按名称模糊搜索函数、变量、类型、宏，并显示其定义位置。
- **FR3.2 调用查询**：给定函数名，可查询其调用者列表和被调用者列表。
- **FR3.3 引用查询**：给定符号（变量、类型、宏），可查询所有引用该符号的位置。
- **FR3.4 文件包含查询**：给定文件，可查询哪些文件包含了它，以及它包含了哪些文件。
- **FR3.5 路径查询**：给定两个函数（或系统调用入口与目标函数），可查询它们之间的调用路径（可能存在多条）。

#### 3.4 可视化分析
- **FR4.1 调用图**：
  - 以函数为节点，调用关系为边，展示函数级调用图。
  - 支持按模块、文件筛选节点，支持展开/折叠子图。
  - 支持节点着色区分类型（如普通函数、系统调用、回调函数）。
  - 支持显示节点属性（函数名、文件、行号、复杂度等）。
  - 支持布局算法（层次布局、力导向布局等）。
- **FR4.2 模块依赖图**：
  - 用户可自定义模块（基于目录、文件列表或正则表达式）。
  - 自动聚合模块内符号，计算模块间依赖（基于调用和包含关系），展示模块级依赖图。
  - 支持显示依赖强度（如边的粗细表示调用次数）。
  - 检测并高亮循环依赖。
- **FR4.3 文件依赖图**：
  - 以文件为节点，包含关系为边，展示文件间的依赖关系。
  - 支持按目录聚合，展示目录级依赖。
- **FR4.4 IPC/内核调用路径图**：
  - 针对 Linux 内核，识别系统调用（如 `SYSCALL_DEFINE` 宏）并作为入口节点，追踪到具体驱动或内核函数，生成调用路径图。
  - 针对 OpenHarmony，识别 Binder 通信、软总线等 IPC 机制，展示服务注册与调用路径。
- **FR4.5 图交互**：
  - 支持缩放、平移、节点拖拽。
  - 支持点击节点查看详细信息，并跳转到对应源码位置。
  - 支持框选节点进行局部放大或导出。
  - 支持保存图为图片（PNG/SVG）或导出图数据（JSON/GraphML）。

#### 3.5 代码编辑器集成
- **FR5.1 编辑器嵌入**：集成代码编辑器（支持语法高亮、代码折叠、自动补全等），用于查看源码。
- **FR5.2 双向跳转**：
  - 在图节点上双击或右键菜单选择“跳转到源码”，编辑器打开对应文件并定位到符号定义位置。
  - 在编辑器中选择符号（或光标置于符号上），右键菜单或快捷键可触发“显示调用图”、“显示调用者”、“显示被调用者”，在图中高亮相关节点。
- **FR5.3 符号信息悬浮提示**：在编辑器中将鼠标悬停在符号上，显示简要信息（如函数签名、定义位置、调用次数等）。

#### 3.6 专用分析功能
- **FR6.1 Linux 内核分析**：
  - 识别系统调用表（`SYSCALL_DEFINE0`~`SYSCALL_DEFINE6`），记录系统调用号与函数名对应关系。
  - 识别 `ioctl` 命令码（通过宏定义），建立设备文件、命令与处理函数的关联。
  - 支持从系统调用入口追踪到具体驱动函数。
- **FR6.2 OpenHarmony 分析**：
  - 识别 Binder 服务端接口（继承 `OHOS::IPC::IRemoteBroker` 的类），记录其方法。
  - 追踪服务注册（`AddService`）和服务获取（`GetService`）调用，建立服务名与服务接口的映射。
  - 分析客户端 `Transact` 到服务端 `OnRemoteRequest` 的调用路径。
  - 识别软总线相关 API，分析分布式通信调用链。
- **FR6.3 自定义规则**：允许用户通过配置文件定义新的 IPC 识别模式（正则表达式或 AST 模式），以扩展对其他 IPC 机制的支持。

#### 3.7 AI 智能助手
- **FR7.1 自然语言问答**：用户可通过聊天界面输入问题，AI 结合当前项目的分析结果（如符号、调用图、依赖）给出回答。
- **FR7.2 上下文感知**：
  - AI 能够感知当前用户正在查看的文件、选中的符号，将其作为上下文的一部分。
  - 用户可手动指定分析范围（整个项目、当前模块、当前文件）。
- **FR7.3 技能调用**：
  - AI 支持通过函数调用方式主动查询平台功能（如获取函数调用者、查找调用路径等），并将结果整合到回答中。
  - 技能集至少包括：符号搜索、调用查询、路径查询、文件依赖查询、模块依赖查询。
  - 技能的定义遵循开放标准（如 OpenAI Function Calling 格式）。
- **FR7.4 多 Agent 协同**：
  - 支持多个 AI Agent 分工协作（如规划 Agent 拆解复杂问题、执行 Agent 调用技能、验证 Agent 检查答案一致性）。
  - Agent 间通信机制可配置，支持串行或并行执行。
- **FR7.5 MCP 协议支持**：
  - 支持作为 Model Context Protocol 服务器，将代码分析数据（如符号、调用关系）作为资源暴露，供外部 MCP 客户端（如 Claude Desktop）接入。
  - 支持作为 MCP 客户端，调用外部 MCP 服务获取额外上下文（如技术文档）。
- **FR7.6 对话历史管理**：保存用户与 AI 的对话历史，支持上下文延续，可清空或导出对话。
- **FR7.7 AI 配置**：允许用户选择 AI 后端（本地模型/云端 API），配置 API 密钥、模型参数等。

### 4. 非功能需求

#### 4.1 性能
- **NFR1.1 解析速度**：对于百万行级项目（如 Linux 内核），全量解析应在可接受时间内完成（例如 30 分钟内），并支持后台解析不阻塞 UI。并行解析是实现该目标的关键手段。
- **NFR1.2 查询响应**：符号搜索、调用查询等操作应在秒级内返回结果。
- **NFR1.3 图渲染**：对于节点数超过 1000 的图，应支持懒加载或聚合显示，避免界面卡顿。
- **NFR1.4 内存占用**：解析大型项目时，内存占用应控制在合理范围（如不超过 8GB），避免崩溃。
- **NFR1.5 远程解析与传输性能**：远程 Agent 的解析性能应充分利用远程服务器的硬件资源。索引数据回传应高效压缩，百万行项目的索引数据传输应在合理时间内完成。SSH 连接应支持复用和保活。

#### 4.2 可用性
- **NFR2.1 界面直观**：主要功能（项目管理、图浏览、代码查看）应布局清晰，操作符合用户习惯。
- **NFR2.2 帮助文档**：提供用户手册，包括功能介绍、操作指南、常见问题。
- **NFR2.3 错误提示**：解析失败、查询无结果、远程同步失败等应有明确提示，并给出可能的原因和建议。
- **NFR2.4 国际化**：支持中文和英文界面，可根据系统语言切换。
- **NFR2.5 认证管理**：远程服务器的 SSH 认证信息应提供友好的输入界面，支持保存和更新，且避免敏感信息泄露。

#### 4.3 兼容性
- **NFR3.1 操作系统**：支持 Windows 10/11、macOS 12+、主流 Linux 发行版（Ubuntu 20.04+，Fedora 等）。
- **NFR3.2 源码环境**：支持 C/C++ 标准（C11/C++17 等），能正确处理常见编译器扩展（GCC、Clang）。
- **NFR3.3 编译数据库**：兼容标准 `compile_commands.json` 格式，以及内核生成的 `compile_commands.json`。
- **NFR3.4 远程协议**：以 SSH 为核心远程访问协议，支持 SSH 密钥认证、密码认证、跳板机（ProxyJump）、端口转发等。支持通过代理服务器连接远程服务器。
- **NFR3.5 远程服务器兼容性**：远程 Agent 需支持主流 Linux 发行版（Ubuntu 18.04+、CentOS 7+、Fedora 等），远程服务器需安装有 LLVM/Clang 开发库或由 Agent 自带静态链接的 LLVM/Clang 库。

#### 4.4 可靠性
- **NFR4.1 数据持久性**：项目配置和解析索引应持久化存储，应用重启后可快速恢复。
- **NFR4.2 崩溃恢复**：解析过程中若崩溃，重启后应能恢复部分索引或提示重新解析。
- **NFR4.3 数据一致性**：增量解析时，应确保索引更新原子性，避免脏数据。并行解析时应妥善处理并发写入，确保最终一致性。
- **NFR4.4 网络容错**：SSH 连接中断时应支持自动重连。远程解析过程中网络断开，Agent 应继续在远程服务器上完成解析，待连接恢复后回传结果。索引数据传输支持断点续传。
- **NFR4.5 认证安全**：存储的认证信息应加密（如使用操作系统密钥链或加密库），避免明文存储。

#### 4.5 安全性
- **NFR5.1 本地数据安全**：所有分析数据存储在本地，不上传用户源码（除非用户主动配置 AI 云端服务）。
- **NFR5.2 AI 隐私**：若使用云端 AI，应提醒用户可能发送代码片段至外部服务，并提供匿名化选项。
- **NFR5.3 远程访问安全**：强制使用 SSH 加密通道，支持 SSH 密钥认证（推荐）和密码认证。Agent 与桌面端之间的所有通信均通过 SSH 隧道加密传输。

#### 4.6 可扩展性
- **NFR6.1 插件机制**：支持通过插件扩展新的分析规则、可视化布局、AI 技能、远程协议等。
- **NFR6.2 开放 API**：提供内部数据访问接口，方便高级用户或脚本调用。

### 5. 约束条件
- 应用应为桌面端独立应用，无需依赖网络（除 AI 云端服务和 SSH 远程连接外）。
- 远程 Agent（`cct-agent`）应为独立的命令行可执行文件，可通过 SSH 自动部署到远程服务器，无需在远程服务器上预装额外依赖（Agent 可静态链接或自带必要库）。
- 开发语言：前端使用 TypeScript/JavaScript，后端使用 Rust（Tauri 框架）。
- 遵循开源许可证（如 GPLv3 或 MIT），具体待定。

### 6. 附录

#### 6.1 术语表
| 术语 | 解释 |
|------|------|
| 编译数据库 | `compile_commands.json`，记录项目中每个文件的编译命令。 |
| 符号 | 代码中的命名实体，如函数、变量、类型、宏。 |
| 调用图 | 以函数为节点，调用关系为边的有向图。 |
| 模块 | 用户自定义的逻辑分组，可对应源码中的目录或文件集合。 |
| IPC | 进程间通信（Inter-Process Communication）。 |
| MCP | Model Context Protocol，模型上下文协议，用于 AI 与数据源交互。 |
| Agent | 人工智能体，能够自主调用工具、推理决策的 AI 单元。 |
| 远程项目 | 源码存储于远端服务器的项目，通过 SSH 连接远程服务器，由远程 Agent 就地解析，索引数据回传本地。 |
| 远程 Agent | 部署在远程服务器上的轻量级解析守护程序（`cct-agent`），负责执行源码解析并生成索引。 |
| SSH 隧道 | 桌面应用与远程 Agent 之间的加密通信通道，基于 SSH 协议。 |

#### 6.2 需求优先级定义
- **P0（必需）**：产品核心功能，必须实现。
- **P1（重要）**：增强用户体验，建议实现。
- **P2（可选）**：高级特性，可根据资源情况取舍。

（优先级可在后续细化需求时标注）
